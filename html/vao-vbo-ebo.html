01 · Mental Model

先建立正确的分工

把 GPU 想象成装配车间：VBO 是原料仓库，EBO 是订单上的取货编号，VAO 是告诉工人“每几个字节代表什么、去哪个仓库取”的说明书。

Buffer · 数据

VBO：仓库

保存一段上传到 OpenGL buffer object 的原始字节。可以放位置、颜色、法线、纹理坐标，OpenGL 不会自动理解它们。

关键：VBO 有数据，但没有数据布局的语义。
Array · 配置

VAO：说明书

记录启用了哪些 vertex attribute，以及每个属性的分量、类型、步长、偏移和数据来源；也记录当前 EBO 绑定。

关键：VAO 通常不保存顶点数据本身。
Element · 索引

EBO：取货单

也是 buffer object，但里面存放的是顶点索引。它让多个三角形复用同一个顶点，避免重复存储相同数据。

关键：EBO 决定按什么编号取顶点，不描述顶点格式。

03 · Draw Calls

Arrays 与 Elements 的差别

两种调用都会使用 VAO 解释顶点；差别在于顶点编号是隐式连续生成，还是显式从 EBO 读取。

glDrawArrays：顺序取

glDrawArrays(GL_TRIANGLES, 0, 6) 依次读取 0～5 号顶点。画两个三角形时，共享的角也需要在 VBO 中重复一份。

ABCCDA
VAO → VBO 顶点 0, 1, 2, 3, 4, 5
不读取 EBO

glDrawElements：按索引取

glDrawElements(..., 6, GL_UNSIGNED_INT, 0) 从 EBO 读取六个编号，再去 VBO 找对应顶点。共享顶点只存一次。

0:A1:B2:C3:D
EBO: 0, 1, 2, 2, 3, 0
VBO 只需 4 个顶点

05 · Common Traps

最容易混淆的三件事

判断一个说法是否正确，可以始终追问两个问题：数据实际存在哪里？解释这段数据的状态记录在哪里？

“VAO 保存了全部顶点”

不准确。顶点字节通常在 VBO；VAO 保存属性格式、启用状态、VBO 关联和 EBO 绑定。

“绑定 VBO 后 GPU 就懂布局”

不准确。必须在 VAO 绑定时调用 glVertexAttribPointer，说明分量、类型、stride 和 offset。

“EBO 是独立全局状态”

在 Core Profile 的常见用法中，GL_ELEMENT_ARRAY_BUFFER 绑定属于当前 VAO 状态；切换 VAO 也会切换它记录的 EBO。

07 · Final Model

最后只记住这三句

当你回到 4-shader/main.cpp 时，按下面三句逐行对照，VAO/VBO/EBO 就不再是孤立 API。

VBO：“我的顶点原始字节放在哪里？”
VAO：“这些字节如何解释，并从哪个 VBO 读取？”
EBO：“这次绘制按什么编号顺序复用顶点？”
